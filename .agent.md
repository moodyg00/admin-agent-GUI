# .agent.md 

## Project: proto-2 + admin-agent-GUI Business Suite

**Vision**: Production-grade business operating system where humans and AI agents collaborate seamlessly on any startup or business. Two apps share the same Postgres database via Prisma:
- proto-2: Public website + simple admin backend (CRUD, business logic)
- admin-agent-GUI: Deep agentic workspace (tasks, memory, content studio, startup lab)

**Tech Stack (preferred)**:
- Next.js 15.3 + React 19 + TypeScript
- Prisma 7 + PostgreSQL (shared DATABASE_URL)
- coss ui components (preferred — see docs/coss-ui.md)
- Tailwind v4, AI SDK, Supabase (auth), Zod

**Core Rules for Grok**:
1. Prefer coss ui components for all UI. You may use shadcn/ui only if it clearly improves the UI/UX and you provide a clear explanation of the specific improvement and why coss was insufficient.
2. Use shared Prisma schema (full business models: User with ai_agent type, Lead, Opportunity, WorkOrder, Invoice, Ad, Integration, etc.).
3. Admin-agent-GUI performs heavy read/write via tRPC or direct Prisma client (initially same DB).
4. Never use generic UI — every screen must feel premium, agent-native.
5. Update these docs when architecture changes.
6. **CRITICAL PROTECTION**: These instruction files (.agent.md, docs/architecture.md, docs/plan.md, docs/context.md, docs/coss-ui.md) are **human-only**. Agents and Grok must **never** edit them.
7. **Agent Communication Protocol**: Every agent **must** read `agent-comms.md` at the start of every task and use the exact HANDOFF format when passing work. The Supervisor enforces this rule.

**Current State**: proto-2 is early Next.js refactor of Laravel with massive Prisma schema. admin-agent-GUI is empty starter.

**Priorities**: See docs/plan.md

**How to Edit**: Use GitHub tools or direct edits. Always commit with clear message. Test UI components before pushing.

**Safety**: Sandbox all agent tool calls. Human approval for production changes.
